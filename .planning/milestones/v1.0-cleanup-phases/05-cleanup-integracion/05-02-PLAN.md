**Output:** `GameRecords.tsx` consume `useGames().fetchAchievements`; misma UX en éxito; en fallo transitorio recupera con un retry; el `.catch(() => {})` directo desaparece.

```ts
const fetchAchievements = useCallback(async (gameId: string): Promise<AchievementsByPlayerDTO | null> => {
  try {
    return await triggerAchievements(gameId)
  } catch {
    // D-09: one retry
    try {
      return await triggerAchievements(gameId)
    } catch {
      // D-10: silent failure — achievements will be calculated eventually (reconciler or next game)
      console.warn('Failed to load achievements after retry')
      return null
    }
  }
}, [])

return { submitting, submitError, submitGame, fetchRecords, fetchAchievements }
```

Notas clave:
- `fetchAchievements(gameId)` retorna `Promise<AchievementsByPlayerDTO | null>`.
- Devuelve `null` cuando ambos intentos fallan (D-10: silent failure deliberado, achievements son no-críticos).
- El `console.warn` ya está dentro del hook — no replicarlo en el caller.
- El callsite actual en GameRecords.tsx aplica una guarda `hasAny` sobre `Object.values(data.achievements_by_player).some(list => list.length > 0)` — esa lógica debe preservarse íntegra cuando `fetchAchievements` resuelve a un valor no-null.

GameRecords.tsx callsite actual a reemplazar (líneas 46-54):
```tsx
triggerAchievements(gameId)

    if (hasAny) {
      setAchievements(data)
      setShowAchievementModal(true)
    }
  })
  .catch(() => {})   // silent failure — achievements are non-critical
```
